# RFID 开源系统 4.0 — 项目功能报告

## 一、项目概览

本项目是一个基于 **STM32F103C6T6** 微控制器的嵌入式 RFID 读卡与电机控制系统，采用 **FreeRTOS** 实时操作系统进行多任务调度。项目使用 STM32CubeMX 生成 HAL 库底层初始化代码，并以 **PlatformIO** 和 **Keil MDK-ARM** 双工具链构建。

**核心功能**：通过 RFID 射频模块读取 IC 卡的 UID 及数据块内容，驱动语音播报模块播报卡内存储的汉字编码，同时控制直流电机的 PWM 调速运转（支持缓启动、定时停止及特定时间段减速）。

| 属性 | 值 |
|------|-----|
| 主控芯片 | STM32F103C6T6A (Cortex-M3, 72MHz) |
| Flash / RAM | 32KB / 10KB |
| 实时系统 | FreeRTOS V10.0.1 (CMSIS-RTOS v1 封装) |
| 构建工具 | PlatformIO + Keil MDK-ARM V5.32 |
| HAL 版本 | STM32Cube FW_F1 V1.8.5 |
| 原作者 | 一点创绘 (卢佳威) / 优灵科技 |
| 作者店铺 | https://ydch.taobao.com |

## 四、RFID 读卡核心功能实现详解

RFID 读卡功能是整个项目最核心的业务逻辑。它不依赖第三方 RFID 库（如 MFRC522），而是通过串口直接与一个外购的 RFID 读卡模块进行私有协议通信，由 STM32 向模块发送指令、接收并解析响应数据。

整个读卡流程涉及三个层次：
- **硬件抽象层**（[Card.c](hardware/rfid_card/Card.c)）：构造指令帧、发送、接收解析
- **中断服务层**（[Card.c](hardware/rfid_card/Card.c) `HAL_UART_RxCpltCallback`）：逐字节驱动接收状态机
- **应用任务层**（[rfid_task.c](Task/rfid_task.c)）：业务状态机，决定何时读卡、如何处理数据

---

### 4.1 数据结构的定义

在读卡之前，先了解三个核心数据结构：

#### 4.1.1 卡片数据体 `CARD`

```c
typedef struct _CARD {
    unsigned int   Type;           // 卡类型
    unsigned long  UID;            // 4字节卡号
    unsigned long  Value;          // 充值/扣款后的余额
    unsigned char  KeyA[6];        // 密钥A
    unsigned char  KeyB[6];        // 密钥B
    unsigned char  BlockData[16];  // 最近一次读块的16字节数据
} CARD;
```

`CARD` 结构体承载一张卡的所有信息。实际使用中主要用到 `UID`（卡号）、`BlockData`（块数据）和 `Value`（数值操作结果）。

#### 4.1.2 指令缓冲区 `CMD`

```c
typedef struct _CMD {
    unsigned char ReceiveBuffer[32];  // 接收缓冲区
    unsigned char SendBuffer[32];     // 发送缓冲区
    unsigned char ReceivePoint;       // 接收指针（未使用）
    unsigned char Code;               // 当前响应的指令码
    unsigned char block_data[16];     // 读块后解析出的块数据副本
} CMD;
```

`CMD` 是一个全局单例（`Cmd`），所有指令的发送和接收共用这块缓冲区。发送时填充 `SendBuffer`，接收时由状态机写入 `ReceiveBuffer`，`CommandProcess()` 再从中解析出 `UID`、`BlockData`、`Code` 等字段。

#### 4.1.3 任务控制结构体 `rfid_control_t`

```c
typedef struct {
    uint8_t  chinese_data[60];      // 跨区块拼接的中文编码缓冲区
    int32_t  chinese_block_num;     // 当前中文数据已读到的区块偏移
    int32_t  read_block;            // 起始读取的区块号（默认=4）
    uint8_t  wait_time;             // 等待响应的计时（单位ms）
    uint8_t  wait_resend_times;     // 超时重发次数
    uint32_t led_flash_counts;      // LED闪烁计数
    float    led_close_counts;      // LED延迟关闭倒计时
} rfid_control_t;
```

这个结构体由 RFID 任务使用，维护跨区块读取的中间状态。`chinese_data[60]` 最多容纳 60 字节（30 个 GB2312 汉字），`chinese_block_num` 记录当前正在读取第几个数据块偏移。

### 4.2 通信协议详解

RFID 模块通过 USART1 与 STM32 通信，采用一套私有二进制帧协议。

#### 4.2.1 物理层与波特率切换

初始化流程在 [usart.c:32-63](Core/Src/usart.c#L32-L63) 的 `MX_USART1_UART_Init()` 中：

```
1. HAL_UART_Init(&huart1)  → 初始化为 9600 bps
2. SetBound115200()        → 以 9600 发送"切换到 115200"的指令
3. 等待发送完成 (UART_FLAG_TC)
4. huart1.Init.BaudRate = 115200
5. HAL_UART_Init(&huart1)  → 重新初始化为 115200 bps
```

为什么要这样做？因为 RFID 模块出厂默认波特率是 9600，只有收到波特率切换指令后才跳到 115200。`SetBound115200()` 和 `SetBound9600()` 两者的区别仅在于指令载荷中的波特率参数：

| 函数 | 指令载荷关键字节 | 目标波特率 |
|------|-----------------|-----------|
| `SetBound115200()` | `0x00 0x01 0xC2` | 115200 |
| `SetBound9600()` | `0x00 0x00 0x25`  | 9600 |

#### 4.2.2 数据帧结构

发送帧和接收帧采用相同的封装格式：

```
┌────────┬──────────────────────────┬──────────┐
│ 0x7F   │ 载荷（转义后）           │ 变长      │
│ 帧头   │ Len + Addr + Cmd + Data  │          │
└────────┴──────────────────────────┴──────────┘
```

**转义规则**：载荷中出现 `0x7F` 时，扩展为两个字节 `0x7F 0x7F` 发送，接收端遇到连续的 `0x7F 0x7F` 时合并为一个 `0x7F`。

#### 4.2.3 载荷内部结构

以读卡号指令 `ReadCard()` 为例，发送载荷为：

```
偏移  内容        说明
[0]   0x03        载荷总长度（含自身和校验，3字节）
[1]   0x00        模块地址（固定为0，广播地址）
[2]   0x10        指令码：读卡号
[3]   Checksum    前3字节的XOR异或值
```

以读块指令 `ReadBlock(block)` 为例：

```
偏移  内容        说明
[0]   0x04        载荷总长度
[1]   0x00        模块地址
[2]   0x11        指令码：读块
[3]   block       目标区块号
[4]   Checksum    前4字节XOR
```

以写块指令 `WriteBlock(block, blockData)` 为例（最长指令）：

```
偏移    内容        说明
[0]     0x14        载荷总长度 = 20
[1]     0x00        模块地址
[2]     0x12        指令码：写块
[3]     block       目标区块号
[4..19] blockData   16字节要写入的数据
[20]    Checksum    前20字节XOR
```

#### 4.2.4 XOR 校验算法

```c
unsigned char CheckSum(unsigned char *dat, unsigned char num) {
    unsigned char bTemp = 0, i;
    for(i = 0; i < num; i++) {
        bTemp ^= dat[i];  // 逐字节异或
    }
    return bTemp;
}
```

校验范围是载荷的第 `[0]` 到第 `[len-1]` 字节（即长度字段到数据字段末尾），校验值放在载荷最后一个字节。接收端对 `ReceiveBuffer[0]` 到 `ReceiveBuffer[ReceiveBuffer[0]-1]` 做 XOR 后与 `ReceiveBuffer[ReceiveBuffer[0]]` 比对，一致则通过。

### 4.3 指令发送：`UartSendCommand()` 帧封装

```c
void UartSendCommand(uint8_t *buff, uint8_t cnt) {
    uint8_t data[20];
    uint8_t num = 0;

    data[0] = 0x7F;                    // 帧头
    for(i = 0; num < cnt + 1; i++) {
        data[i+1] = buff[i];           // 逐字节拷贝
        num++;
        if(buff[i] == 0x7F) {          // 遇到特殊字节 → 转义
            i += 1;
            data[i+1] = 0x7F;          // 插入一个额外的 0x7F
        }
    }
    num = i + 1;                       // 最终帧长度
    HAL_UART_Transmit(&CARD_HAL_USARTx, data, num, 0xffff);
}
```

这个函数是**同步阻塞发送**。注意 `cnt` 参数是载荷字节数（含长度字段和校验码），而 `num` 最终统计的是加上帧头和转义字节后的实际发送字节数。

例如 `ReadCard()` 载荷为 3 字节 `{0x03, 0x00, 0x10, 0x13}`，不含 `0x7F`，最终发送 4 字节 `{0x7F, 0x03, 0x00, 0x10, 0x13}`。

### 4.4 指令接收：`UartReceiveCommand()` 状态机

这是整个读卡功能中最复杂的函数。它被串口中断回调**每收到一个字节就调用一次**，内部维护静态变量实现增量式状态机解析。

#### 4.4.1 状态机设计

```
                    ┌──────────────────────┐
                    │   start_receive = 0  │  ← 初始状态：等待帧头
                    │   等待 0x7F          │
                    └──────────┬───────────┘
                               │ 收到 0x7F
                               ▼
                    ┌──────────────────────┐
                    │   start_receive = 1  │
                    │   len = 0, i = 0     │
                    └──────────┬───────────┘
                               │
                    ┌──────────▼───────────┐
                    │   len == 0 ?         │
                    │   首次进入：读长度    │
                    └──────────┬───────────┘
                               │ data < 0x7F → len = data
                               ▼
                    ┌──────────────────────┐
                    │   len > 1 ?          │
                    │   逐字节收载荷       │
                    │   len--; buf[i++]    │
                    └──────────┬───────────┘
                               │ len 递减到 1
                               ▼
                    ┌──────────────────────┐
                    │   len == 1 (最后一字节)│
                    │   重置状态机          │
                    │   判断响应类型        │
                    │   return 结果码       │
                    └──────────────────────┘
```

#### 4.4.2 核心代码逐段分析

```c
uint8_t UartReceiveCommand(uint8_t data) {
    static uint8_t start_receive = 0;  // 是否已找到帧头
    static uint8_t len = 0;            // 剩余待收字节数
    static unsigned char i = 0;        // 写入缓冲区的索引

    // ── 阶段1：等待帧头 0x7F ──
    if(start_receive == 0) {
        if(data == 0x7F) {
            start_receive = 1;         // 找到帧头，进入接收态
            len = 0;
            i = 0;
        }
        return 0;                      // 未找到帧头，返回无结果
    }
    // ── 阶段2：已找到帧头 ──
    else {
        if(len == 0) {                 // 下一字节 = 载荷长度
            if(data < 0x7F) {
                len = data;            // 设置待收字节数
            }
        }
        else if(len > 1) {             // 收到的是数据载荷（非最后字节）
            len--;
            Cmd.ReceiveBuffer[i++] = data;  // 写入全局接收缓冲
        }
        else {                         // len == 1：载荷最后一字节
            len = 0;
            start_receive = 0;         // 重置状态机，准备下一帧

            // ── 载荷接收完毕，开始解析 ──
```

当状态机走完一轮（`len` 从初值递减到 1，收到最后一个字节后重置），`Cmd.ReceiveBuffer` 中已经有了完整的载荷数据。此时按 `ReceiveBuffer[1]`（响应码）判断数据类型：

```c
            // 响应码 0x90 → 读卡号响应
            if(Cmd.ReceiveBuffer[1] == 0x90) {
                if(Cmd.ReceiveBuffer[2] == 0x00) {
                    return 2;          // 读卡成功
                } else {
                    return 3;          // 读卡失败（无卡等）
                }
            }

            // 响应码 0x91 → 读块响应
            if(Cmd.ReceiveBuffer[1] == 0x91) {
                if(Cmd.ReceiveBuffer[2] == 0x00) {
                    // 成功：提取 16 字节块数据
                    for(i = 0; i < 16; i++) {
                        Cmd.block_data[i] = Cmd.ReceiveBuffer[9 + i];
                    }
                } else {
                    return 3;
                }
            }

            // 响应码 0xAC → 设置波特率响应
            else if(Cmd.ReceiveBuffer[1] == 0xAC) {
                if(Cmd.ReceiveBuffer[2] == 0x00) {
                    // 成功，无额外处理
                } else {
                    return 3;
                }
            }
            return 1;                  // 通用"收到正确数据"
        }
        return 0;                      // 数据未收完，继续等待
    }
}
```

#### 4.4.3 返回值含义

| 返回值 | 含义 | 触发条件 |
|--------|------|---------|
| `0` | 尚无结果 | 状态机未完成一帧的接收 |
| `1` | 收到正确数据帧 | 任何未知响应码的帧接收完毕（不含卡检测） |
| `2` | 读卡成功 | 收到 `0x90 0x00` 响应（线圈内有卡） |
| `3` | 数据错误 | `0x90` 或 `0x91` 或 `0xAC` 的状态码非 `0x00`（通常为 `0xFF` = 无卡） |

### 4.5 中断回调：连接硬件与任务

```c
uint8_t temp_res;
extern rfid_control_t rfid_control;

void HAL_UART_RxCpltCallback(UART_HandleTypeDef *huart) {
    // 每次收到1字节时进入此回调
    temp_res = UartReceiveCommand(card_res);   // 喂入状态机

    if(temp_res == 1) {                        // 收到完整数据帧
        card_res_flag = CARD_FLAG_RESDATA;
    }
    else if(temp_res == 2) {                   // 读卡成功（线圈内有卡）
        if(card_res_flag != CARD_FLAG_LEDLIGHT) {
            card_res_flag = CARD_FLAG_EXIST;   // 通知任务：有卡
        }
    }
    else if(temp_res == 3) {                   // 数据错误（通常是无卡）
        if(card_res_flag == CARD_FLAG_LEDLIGHT) {
            rfid_control.led_close_counts--;   // LED延迟关闭倒计时
            if(rfid_control.led_close_counts <= 0) {
                card_res_flag = CARD_FLAG_NONE;
            }
        } else {
            card_res_flag = CARD_FLAG_NONE;    // 回到空闲状态
        }
    }

    HAL_UART_Receive_IT(&huart1, (uint8_t *)&card_res, 1);  // 重新开启中断，接收下一字节
}
```

关键设计要点：
1. **每次只接收 1 字节**：`HAL_UART_Receive_IT(&huart1, &card_res, 1)` 注册单字节中断接收。回调末尾**重新注册**，形成"收一个字节 → 回调 → 再注册收下一个字节"的循环。
2. **全局标志 `card_res_flag`**：这是中断上下文与任务上下文之间唯一的通信方式。中断只负责设置标志，RFID 任务的主循环轮询检查标志并采取行动。
3. **`CARD_FLAG_LEDLIGHT` 状态下的特殊处理**：当卡片处于 LED 点亮延迟关闭阶段，无卡响应不立即变为 `NONE`，而是递减计数器，目的是让 LED 延迟关闭期间不因卡片短暂离开线圈而误复位。

#### 4.5.1 标志位状态转移图（中断侧）

```
                 ┌─────────────────────────────────────┐
                 │          card_res_flag               │
                 └─────────────────────────────────────┘
                                │
         temp_res==2            │           temp_res==3
         (有卡)                 │            (无卡/错误)
                                ▼
    ┌─────────────────────────────────────────────┐
    │  card_res_flag != LEDLIGHT ?                │
    │    YES → CARD_FLAG_EXIST (通知任务)          │
    │    NO  → 保持 LEDLIGHT (被忽略)              │
    └─────────────────────────────────────────────┘
                                │
                                ▼
    ┌─────────────────────────────────────────────┐
    │  card_res_flag == LEDLIGHT ?                │
    │    YES → led_close_counts--                 │
    │          若归零 → CARD_FLAG_NONE             │
    │    NO  → 直接 CARD_FLAG_NONE                 │
    └─────────────────────────────────────────────┘
```

### 4.6 响应数据解析：`CommandProcess()`

虽然本项目在 `RFID_Task` 中直接使用了 `Cmd.block_data`（读块）和 `Card.UID`（读卡号），但 `CommandProcess()` 函数展示了完整的响应解析逻辑：

```c
unsigned char CommandProcess(void) {
    unsigned char i, status;
    // 校验：ReceiveBuffer[长度字段] 位置的值应等于前面所有字节的XOR
    if(Cmd.ReceiveBuffer[Cmd.ReceiveBuffer[0]] == CheckSum(Cmd.ReceiveBuffer, Cmd.ReceiveBuffer[0])) {

        // 提取卡号（大端序拼接）
        Card.UID  = Cmd.ReceiveBuffer[9]; Card.UID <<= 8;
        Card.UID |= Cmd.ReceiveBuffer[8]; Card.UID <<= 8;
        Card.UID |= Cmd.ReceiveBuffer[7]; Card.UID <<= 8;
        Card.UID |= Cmd.ReceiveBuffer[6];

        // 按指令码分发处理
        switch(Cmd.ReceiveBuffer[2]) {
            case 0x90: break;                              // 读卡号
            case 0x91:                                     // 读块：提取16字节块数据
                for(i=0;i<16;i++) Card.BlockData[i] = Cmd.ReceiveBuffer[10+i];
                break;
            case 0x95: case 0x96:                          // 充值/扣款：提取4字节余额
                Card.Value = Cmd.ReceiveBuffer[10] << 24 |
                             Cmd.ReceiveBuffer[11] << 16 |
                             Cmd.ReceiveBuffer[12] << 8  |
                             Cmd.ReceiveBuffer[13];
                break;
            // ...
        }
        status = Cmd.ReceiveBuffer[3];  // 状态码：0=成功, 0xFF=无卡, etc.
    } else {
        status = 0xFB;  // 校验失败
    }
    return status;
}
```

**标准响应帧的载荷布局**（以 `0x91` 读块成功为例）：

```
偏移    内容
[0]     载荷长度（含自身，如 0x1A = 26）
[1]     模块地址 = 0x00
[2]     指令码 = 0x91（读块响应）
[3]     状态码 = 0x00（成功）/ 0xFF（无卡）/ 其他
[4..5]  卡类型（2字节）
[6..9]  卡号 UID（4字节，ReceiveBuffer[6]为最低字节）
[10..25] 块数据（16字节）
[26]    Checksum
```

### 4.7 RFID 任务：业务状态机

应用层逻辑位于 `RFID_Task()`（[rfid_task.c:29-167](Task/rfid_task.c#L29-L167)）。它运行在 FreeRTOS 任务上下文中，优先级为 `osPriorityHigh`，以 1ms 为周期轮询 `card_res_flag` 并驱动业务状态转移。

#### 4.7.1 初始化阶段（任务开头，仅执行一次）

```
vTaskDelay(500)                → 等待语音模块上电稳定
if(RFID_SETTING_SPEAK_SPEED):
    printf("<S>3")             → 设置语速为最快
    printf("<V>6")             → 设置音量为最大
    printf("<I>0")             → 保存设置
RFID_Par_Init()                → 控制结构体清零
    chinese_block_num = 0
    read_block = 4
    card_res_flag = CARD_FLAG_EXIST (若 RFID_READ_DATA_WHEN_START=1)
                  或 CARD_FLAG_NONE
```

#### 4.7.2 主循环五个状态详解

```
┌──────────────┐
│ NONE    (0)  │  空闲状态：LED灭，清空数据缓冲区，等待中断回调将标志改为 EXIST
└──────┬───────┘
       │ 中断检测到卡片 → card_res_flag = EXIST
       ▼
┌──────────────┐
│ EXIST   (3)  │  有卡：发送 ReadBlock(read_block + chinese_block_num)，
│              │  然后立即将标志切换为 WAIT
└──────┬───────┘
       │ ReadBlock() 已发出，等待 RFID 模块响应
       ▼
┌──────────────┐
│ WAIT    (2)  │  等待响应：每 1ms 递增 wait_time
│              │  - 20ms 内收到响应 → card_res_flag = RESDATA
│              │  - 20ms 超时        → card_res_flag = EXIST (重发)
│              │  - 重发 2 次仍超时   → card_res_flag = NONE  (放弃)
└──────┬───────┘
       │ 中断回调收到完整数据帧 → card_res_flag = RESDATA
       ▼
┌──────────────┐
│ RESDATA (1)  │  收到数据：根据 chinese_block_num 判断数据完整性
│              │
│  ┌─────────────────────────────────────────────────┐
│  │ chinese_block_num == 0 且 block_data[0]==0      │
│  │   → 第4区块无数据 → chinese_block_num = -3      │
│  │   → card_res_flag = EXIST (回退读第1区块)        │
│  ├─────────────────────────────────────────────────┤
│  │ chinese_block_num == -3 且 block_data[0]==0     │
│  │   → 第1区块也无数据 → card_res_flag = NONE       │
│  ├─────────────────────────────────────────────────┤
│  │ chinese_block_num == -3 且 block_data[0]!=0     │
│  │   → 第1区块有数据 → LED亮 + 语音播报             │
│  │   → card_res_flag = LEDLIGHT                    │
│  ├─────────────────────────────────────────────────┤
│  │ chinese_block_num > 0                           │
│  │   → 使用已拼好的 chinese_data[60]               │
│  │   → 若前2字为"一点"→ 不播报不亮灯                │
│  │   → 否则 → LED亮 + 语音播报                      │
│  │   → card_res_flag = LEDLIGHT                    │
│  ├─────────────────────────────────────────────────┤
│  │ 否则（chinese_block_num 0~3 且数据非零）         │
│  │   → 将 block_data 追加到 chinese_data           │
│  │   → chinese_block_num++                         │
│  │   → card_res_flag = EXIST (继续读下一区块)       │
│  └─────────────────────────────────────────────────┘
└──────┬───────┘
       │ 语音播报完毕后
       ▼
┌──────────────┐
│ LEDLIGHT(4)  │  LED延迟关闭阶段：
│              │  LED保持亮，定期 ReadCard() 检测卡片是否还在
│              │  led_close_counts 被中断回调递减直到 0
│              │  归零后 → card_res_flag = NONE
└──────────────┘
```

#### 4.7.3 中文跨区块读取机制（核心）

项目的中文编码采用 **GB2312 标准**，每个汉字占 2 字节。一张 MIFARE Classic 1K 卡有 16 个扇区，每扇区 4 个块，每个数据块可存 16 字节 = 8 个汉字。`chinese_data[60]` 最多可容纳 60 字节 = 30 个汉字。

读取流程依赖于 `chinese_block_num` 的含义变化：

| `chinese_block_num` 值 | 含义 |
|------------------------|------|
| `0` | 初始状态，尚未开始跨区块读取 |
| `1, 2, 3...` | 正在逐区块拼接：表示已有几块数据写入 `chinese_data` |
| `-3` | 回退模式：第4区块无数据，退回到第1区块读取 |

**第一次读取**（`chinese_block_num = 0` 时检测到卡）：

```
ReadBlock(4 + 0)  →  读取第4区块
↓ 收到数据
block_data[0] != 0 ?  →  有数据：这是官方APP录入的
    chinese_block_num = 1
    将 block_data 写入 chinese_data[0..15]
    继续读取第5区块 (4+1), 第6区块 (4+2)...

block_data[0] == 0 ?  →  无数据：不是官方APP录入的
    chinese_block_num = -3
    回退读取第1区块 (4 + (-3))
```

**回退模式**（`chinese_block_num = -3`）：

因为只回退一次读取一个区块，所以直接判断该区块是否有数据：
- 有数据 → 直接播报（非官方APP，不逐区块拼接）
- 无数据 → 放弃，返回空闲

**正常拼接模式**（`chinese_block_num 从 1 开始递增`）：

每一轮循环：
1. 读取区块 `4 + chinese_block_num`
2. 将 16 字节块数据追加到 `chinese_data[chinese_block_num * 16]`
3. `chinese_block_num++`
4. 继续读下一区块

当某个区块读取的 16 字节全为 `0x00` 时，说明数据结束，此时 `chinese_block_num` 已累计到正值（≥1），跳入"数据完整"分支，使用已拼接好的 `chinese_data` 进行播报。

#### 4.7.4 "一点"品牌标识过滤

```c
if(rfid_control.chinese_data[0]==0xD2 && rfid_control.chinese_data[1]==0xBB
&& rfid_control.chinese_data[2]==0xB5 && rfid_control.chinese_data[3]==0xE3) {
    // 不播报不亮灯
}
```

- `0xD2 0xBB` = GB2312 编码的"一"
- `0xB5 0xE3` = GB2312 编码的"点"

当卡内数据以"一点"（品牌名"一点创绘"的前两个字）开头时，系统静默处理，不触发播报和亮灯。

#### 4.7.5 语音播报的实现

```c
LED_Sta(1);                              // LED 亮
Usartx_SendString(rfid_control.chinese_data);  // 通过 USART2 发送 GB2312 编码
while(!__HAL_UART_GET_FLAG(&HAL_USARTX, UART_FLAG_TC));  // 等待发送完成
vTaskDelay(RFID_DELAY * 1000);           // 延时（默认 800ms），防止连续重复读同一张卡
card_res_flag = CARD_FLAG_LEDLIGHT;      // 进入 LED 延迟关闭阶段
rfid_control.led_close_counts = RFID_LED_LATE_CLOSE_TIME * 20;  // 0.5 × 20 = 10
```

语音合成模块连接在 **USART2**（`huart2`, 9600bps），它接收 GB2312 编码的字节串后自动合成语音输出。每个汉字两个字节，模块内部有字库完成编码到语音的映射。

#### 4.7.6 卡片持续在位检测（LEDLIGHT 状态）

```c
if(card_res_flag == CARD_FLAG_LEDLIGHT) {
    LED_Sta(1);
    ReadCard();            // 持续发送读卡号指令
    vTaskDelay(10);        // 每 10ms 检测一次
}
```

当卡处于 `LEDLIGHT` 状态时，任务每 10ms 发送一次 `ReadCard()`（指令码 `0x10`），RFID 模块返回有卡/无卡。中断回调根据返回值：
- **有卡**（`temp_res == 2`）：因当前是 `LEDLIGHT` 状态，回调直接忽略（不改变状态），保持 LED 亮
- **无卡/错误**（`temp_res == 3`）：每收到一次错误，`led_close_counts--`，直到归零后 `card_res_flag = NONE`，LED 灭

这意味着卡片离开线圈后 LED 不会立即熄灭，而是经过 `RFID_LED_LATE_CLOSE_TIME × 20 ≈ 10` 次无卡检测才熄灭，即约 100ms（10ms × 10 次）的延迟。

### 4.8 完整读卡时序总结

下图展示了一次完整的"卡片靠近 → 读取中文 → 语音播报 → LED 熄灭"的时间线：

```
时刻    事件                             触发者
─────────────────────────────────────────────────────
T0      卡片靠近线圈                     物理事件
T0+?    RFID模块检测到卡，响应ReadCard   中断(isr)
        中断回调: card_res_flag=EXIST
T1      RFID_Task轮询到EXIST             任务(task)
        发送 ReadBlock(4)
        card_res_flag=WAIT
T2      RFID模块响应块数据                中断(isr)
        回调: card_res_flag=RESDATA
T3      RFID_Task轮询到RESDATA            任务(task)
        发现block_data[0]!=0
        将数据追加到chinese_data
        chinese_block_num=1
        card_res_flag=EXIST  → 继续读Block 5
T4~Tn   重复T1~T3，逐区块拼接chinese_data  任务+中断
Tn      读取到全零区块，数据拼接完成       任务(task)
        LED亮 + Usartx_SendString发送GB2312
        等待发送完成
        card_res_flag=LEDLIGHT
Tn+?    卡片离开线圈                     物理事件
Tn+?    中断回调收到无卡→递减计数器       中断(isr)
        led_close_counts归零
        card_res_flag=NONE
        LED_Sta(0) → LED熄灭              任务(task)
```

### 4.9 设计亮点与注意事项

1. **中断与任务解耦**：中断回调只操作 `card_res_flag` 标志位，不做任何耗时操作；业务逻辑完全在任务中处理，符合 RTOS 最佳实践。

2. **单字节接收 + 静态状态机**：`UartReceiveCommand()` 使用 `static` 变量维护跨调用状态，使得每次进入只需处理一个字节，极其轻量，适合在中断上下文中运行。

3. **帧转义处理**：发送端做了 `0x7F` 转义（`UartSendCommand`），但接收端的状态机**未实现转义解义**——它仅用 `len` 直接计数剩余字节数，末尾一字节后直接重置。这意味着如果 RFID 模块在响应数据中转义了 `0x7F`，当前接收端代码无法正确还原。好在 RFID 模块的响应载荷中通常不含 `0x7F`。

4. **超时重发机制**：`wait_time` 和 `wait_resend_times` 实现了简单的超时重发（2 次 × 20ms），防止因串口丢字节导致任务永久卡在 `WAIT` 状态。

5. **内存占用**：`chinese_data[60]` 是任务栈上的大数组（栈大小 1024 字 = 4096 字节），如果未来需要更长的播报文本，需同时增大 `rfid_control_t.chinese_data` 和任务栈大小。

## 五、电机控制与辅助功能

### 5.1 电机控制任务（[motor_control_task.c](Task/motor_control_task.c)）

#### 5.1.1 电位器采样与时间计算

1. 读取 ADC1_IN9 通道（PB1 引脚）20 次取平均值
2. 根据 ADC 值反算电位器电阻值：`res = adc ≥ 4095 ? 0 : 5000 - adc × 1000/(4095 - adc)`
3. 线性映射计算电机停止时间：`stop_time = res > 5000 ? STOP_TIME_MAX : STOP_TIME_MIN + res × (STOP_TIME_MAX - STOP_TIME_MIN) / RES_MAX`

#### 5.1.2 电机控制流程

```
初始化采样 → 延迟等待 (MOTOR_START_LATE_TIME_SETUP)
    │
    ▼
主循环 (1ms 周期)
    │
    ├── 缓启动阶段 (0 ~ MOTOR_START_SLOW_TIME_SETUP ms)
    │     PWM 占空比从 0 线性递增至 MOTOR_MAX
    │
    ├── 正常运行阶段
    │     PWM 占空比 = MOTOR_MAX (默认999)
    │
    ├── 特定时间减速段 (MOTOR_TIME_Start ~ MOTOR_TIME_Stop 秒)
    │     PWM 占空比降至 MOTOR_MIN (默认500)
    │
    ├── 定时停止 (count_ms > stop_time)
    │     PWM 占空比 = 0 (电机关闭)
    │
    └── 最大保护 (count_ms ≥ 1000s)
          PWM 占空比 = 0 (强制关闭)
```

#### 5.1.3 PWM 驱动细节（[PWM.c](hardware/pwm/PWM.c)）

- 定时器 TIM2, 频率 1kHz (72MHz / 72 / 1000)
- 双通道输出：CH1 (PA0), CH2 (PA1)
- `Motor_Control(speed)`：speed = 0 时两通道均输出高电平 (制动)；speed > 0 时 CH1 低电平 / CH2 占空比控制

### 5.2 语音播报模块

- 通过 **USART2** (9600bps) 与语音合成模块通信
- 初始化时发送配置指令：`<S>3` (语速 3)、`<V>6` (音量 6)、`<I>0` (保存设置)
- 播报内容为 UID 块中存储的 GB2312 中文字符编码字节串，直接通过 `Usartx_SendString()` 发送至语音模块

### 5.3 LED 控制（[led.c](hardware/LED/led.c)）

- 同时控制 3 个引脚 (PC13, PB12, PA8)，适配不同硬件版本
- `LED_Sta(1)`：三个引脚均输出低电平 (亮)
- `LED_Sta(0)`：三个引脚均输出高电平 (灭)
- `LED_Toggle()`：三个引脚电平翻转

### 5.4 按键检测（[key.c](hardware/KEY/key.c)）

- 按键引脚：PA11
- `Key_StaRead()`：返回按键状态（0 = 按下, 1 = 松开）
- 预留了 EXTI 外部中断回调的注释代码（PlatformIO 环境下需特殊处理才能识别）

### 5.5 调试支持

- `printf` 通过 `fputc` 重定向至 USART2（[BSP_USART.c:41-45](hardware/USART/BSP_USART.c#L41-L45)）
- [syscall.c](hardware/USART/syscall.c) 提供了 `_write` / `_read` / `_sbrk` 等系统调用实现，支持标准 C 库
- `RFID_DEBUG_MODE` 宏控制调试信息的语音播报

## 八、技术亮点

1. **FreeRTOS 双任务架构**：RFID 处理与电机控制分离，RFID 任务以高优先级 (2) 运行，电机控制以空闲优先级 (-3) 运行，确保卡片识别响应及时
2. **灵活的数据读取策略**：自动检测中文数据存储位置（官方 APP 录入 vs 其他方式），兼容不同区块编排
3. **电机多段控制**：晚启动 → 缓启动 → 全速运行 → 特定时间减速 → 定时停止，完整覆盖应用场景
4. **波特率自适应切换**：RFID 模块从 9600 动态切换至 115200，兼顾兼容性和通信效率
5. **串口帧协议解析**：基于帧头 `0x7F` + 转义机制的可靠数据帧解析状态机
6. **品牌过滤机制**：特定内容不触发展示，满足业务定制需求
7. **用户参数集中管理**：[user_setup.h](Task/user_setup.h) 集中所有可调参数，无需深入代码即可定制行为

| 资源 | 总容量 | 占用估算 |
|------|--------|----------|
| Flash | 32 KB | HAL 库 + FreeRTOS + 应用代码 ≈ 20-25 KB |
| RAM | 10 KB | 堆 (6368B) + 栈 (1024B) + 任务栈 (1024+512) + BSS ≈ 9-10 KB (接近满载) |
| FreeRTOS 堆 | 6368 B | 用于两个任务的 TCB 和栈分配 |

> **注意**：STM32F103C6 仅有 10KB RAM，本项目的内存使用已接近上限。若需扩展功能，建议换用 STM32F103C8 (20KB RAM) 或优化栈/堆配置。
